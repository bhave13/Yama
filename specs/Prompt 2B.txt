## ═══════════════════════════════════════════════

All question objects produced by your module must conform to this schema.
Fields are conditional based on `type` — only include fields relevant to the type.

```javascript
// Universal fields — present on ALL question types
{
  uid:      "num-001-lev3-d2",   // makeUID(type, level, diff) — NEVER hardcoded
  level:    3,                    // Skill level (matches Level N in Topic Analysis)
  diff:     2,                    // Difficulty 1–4 (1=integers, 2=decimals, 3=fractions, 4=complex)
  type:     "NUMERIC",            // NUMERIC | MCQ | MATCH | SPOT_ERROR | EXPLANATION
  q:        "",                   // Question text — HTML supported, MathJax LaTeX supported
  working:  "",                   // Step-by-step solution HTML (see generateSolution rules below)
  img:      "",                   // Inline SVG data URI string, OR "" (empty = centred text layout)
  imgAlt:   "",                   // Accessibility description of the SVG (empty if no SVG)
  hint:     "",                   // Optional scaffolded hint string (empty string if none)

// ── NUMERIC ────────────────────────────────────────────────────────────
// Additional fields for type === "NUMERIC"
{
  a:         "12.5",              // Canonical answer string — COMPUTED in code, never a hardcoded literal
  units:     ["m", "metres"],     // Accepted unit strings — empty array [] if unitless

}

// ── MCQ ───────────────────────────────────────────────────────────────
// Additional fields for type === "MCQ"
{

  // not arbitrary wrong values. Use the misconceptions listed in the Topic Analysis.
}

// ── MATCH ─────────────────────────────────────────────────────────────

{

    { left: "Area of triangle",  right: "½ × b × h" },
  ],
  // MATCH RULE: Pairs must be shuffled before output so left and right
  // columns do not visually align with each other.
}

// ── SPOT_ERROR / STEP ─────────────────────────────────────────────────
// Additional fields for type === "SPOT_ERROR" with subtype "STEP"
{
  subtype: "STEP",

  // Never zero errors. Never two errors.
}

// ── SPOT_ERROR / VALUE ────────────────────────────────────────────────

  subtype:        "VALUE",
  expression:     "A = [π|1] × [r²|2] = [50.24|3] cm²",

  //              The framework renders each token as a clickable highlighted span.
  //              The student clicks the token that contains the error.

  errorExplanation: "The radius is 4 cm, so r² = 16, not 16. The correct area is π × 16 ≈ 50.27 cm².",
  // SPOT_ERROR/VALUE RULE: Exactly ONE token must be the error (correctErrorId).

// ── EXPLANATION ───────────────────────────────────────────────────────
// Additional fields for type === "EXPLANATION"
{
  modelAnswer:      "Because the relationship is linear, doubling the input doubles the output...",

    "Gives a correct numerical example",
```

### LaTeX Escaping Rule
All LaTeX within JavaScript string values must use **double backslashes**.
Single backslashes cause JSON/JS parse errors.

| Correct | Incorrect |
|---|---|
| `"\\frac{1}{2}"` | `"\frac{1}{2}"` |
| `"a^2 + b^2 = c^2"` | fine — no backslash needed |
| `"\\sqrt{25} = 5"` | `"\sqrt{25} = 5"` |

### uid Format
Use `makeUID(type, level, diff)` which returns a string in this format:

```
{type_abbr}-{NNN}-lev{L}-d{D}
```

| type | abbr |
|---|---|
| NUMERIC | num |
| MCQ | mcq |
| MATCH | mat |
| SPOT_ERROR/STEP | sep |
| SPOT_ERROR/VALUE | sev |
| EXPLANATION | exp |

Example: `makeUID("NUMERIC", 3, 2)` → `"num-007-lev3-d2"`
The counter NNN increments across all questions in the module.

### generateSolution(q) Format Rules

Return an HTML string. Do not modify the DOM. Show the answer prominently first, then full working.

| Type | Solution Content |
|---|---|
| NUMERIC | Answer in large/bold text, then each computation step on a separate line |
| MCQ | Correct option label + one-line explanation of why it is correct |
| MATCH | All correct pairs listed clearly |
| SPOT_ERROR/STEP | All steps listed; error step highlighted in red with explanation of the mistake |
| SPOT_ERROR/VALUE | Full expression shown; incorrect token highlighted with explanation of the correct value |

Use HTML elements for structure: `<p>`, `<strong>`, `<span style="color:red">` for error highlighting, `<br>` for line breaks within working. All mathematical expressions must use LaTeX notation so MathJax can render them.

## MATHEMATICAL ACCURACY REQUIREMENTS

These are non-negotiable. Violating any of these makes the module unusable.

- **All answers computed programmatically** — never use a hardcoded answer string (e.g. `a: "12.5"` must be derived from the generated parameters, not typed in)
- **The value of `a` must be independently derived** in code from the generated random parameters — compute it the same way a student would
- **No question may have an ambiguous or non-unique correct answer**
- **SPOT_ERROR questions must contain exactly one error** — never zero, never two
- **MCQ distractors must represent common student misconceptions** — use the misconceptions listed in the Topic Analysis; never use arbitrary wrong values
- **MATCH pairs must be shuffled** before output — left and right columns must not visually align
- **Working solutions must show every computation step** on a separate line — no skipped steps
- **Number ranges must produce clean answers** appropriate for the year group — avoid ugly recurring decimals at low difficulty levels
- **LaTeX used for all mathematical expressions** in question text, working, and solutions — always with double backslashes in string values

## QUESTION GENERATION RULES

**Pool-and-Remove Pattern:**
For each level, maintain a pool of context scenarios (e.g. object names, settings, narratives). `pickAndRemove()` selects from the pool so no context repeats within a session. When the pool is exhausted, reset it silently (refill from the original array).

**Type Mix:**
Apply `question_type_mix` proportionally within each level. For a level with 10 questions and a mix of 30% NUMERIC / 20% MCQ / 15% MATCH / 20% SPOT_ERROR / 15% EXPLANATION, generate approximately 3 NUMERIC, 2 MCQ, 2 MATCH, 2 SPOT_ERROR, 1 EXPLANATION. Round sensibly — the mix is a guideline, not a strict quota.

**Difficulty Distribution:**
Spread difficulties 1–4 across the `questions_per_level` questions within each level. Do not generate all questions at the same difficulty.

**Skill Fidelity:**
Each level's questions must test the single skill defined for that level in the Topic Analysis. Do not let questions drift into testing skills from other levels.

**Context Problems:**
At the higher skill levels (Context Problem tier), wrap the question in a real-world NZ narrative. The mathematical structure remains NUMERIC or EXPLANATION. Use the context themes from Section 4 of the Topic Analysis.

**Terminology / Recall Questions:**
At Levels 1–2 (Pre-req Check, Facts), MCQ and MATCH questions may test mathematical vocabulary: correct spelling of terms, matching terms to definitions, identifying symbols. These are valid and pedagogically important (Rosenshine: retrieval practice).

**Example Problems:**
Use the example problems in Section 2 for style calibration — register, sentence structure, number ranges, diagram style. Do NOT copy any question verbatim.

**SVG Diagrams (if include_svg is Yes):**
Generate diagrams as inline `data:image/svg+xml` URI strings. Set `q.img` to the URI and `q.imgAlt` to an accessibility description. When `q.img === ""`, `renderFront()` must apply a CSS class that centres the question text both horizontally and vertically.
