## PURPOSE OF THIS PROMPT

This prompt produces a **human-reviewable Markdown document** that fully specifies every
question in the YAMA topic module before any code is written.

The document is the **single source of truth** for the subsequent code-generation step
(Prompt 2B-Generate). A teacher and developer review it together, sign it off, and then
pass it — unmodified — as the sole input to Prompt 2B-Generate.

**Do not generate any JavaScript in this step.**

## SECTION 3 — YAMA QUESTION SCHEMA v1.2 (STATIC)
## ═══════════════════════════════════════════════

All question objects produced by the final module must conform to this schema.
Fields are conditional based on `type` — only include fields relevant to the type.

```javascript
// Universal fields — present on ALL question types
{
  uid:      "num-001-lev3-d2",   // makeUID(type, level, diff) — NEVER hardcoded
  level:    3,                    // Skill level (matches Level N in Topic Analysis)
  diff:     2,                    // Difficulty 1–4 (1=integers, 2=decimals, 3=fractions, 4=complex)
  type:     "NUMERIC",            // TEXT | NUMERIC | MCQ | MATCH | SPOT_ERROR | EXPLANATION
  q:        "",                   // Question text — HTML supported, MathJax LaTeX supported
  working:  "",                   // Step-by-step solution HTML
  img:      "",                   // Inline SVG data URI string, OR "" (empty = centred text layout)
  imgAlt:   "",                   // Accessibility description of the SVG (empty if no SVG)
  hint:     "",                   // Optional scaffolded hint string (empty string if none)
  ncea:     { standard: "AS91030", ao: "N01" },
}
// NUMERIC — additional fields
{
  a:         "12.5",              // Canonical answer — COMPUTED in code, never a hardcoded literal
  units:     ["m", "metres"],     // Accepted unit strings — empty array [] if unitless
  tolerance: 0.02,                // Accepted margin of error as a decimal fraction
}
// TEXT — additional fields
{
  a:         "trapezium",         // Answer string — a hardcoded literal is acceptable here
}
// MCQ — additional fields
{
  options:       ["10", "12.5", "15", "20"],  // Always exactly 4 options
  correctOption: 1,                            // 0-based index into options[]
}
// MATCH — additional fields
{
  pairs: [
    { left: "Area of rectangle", right: "l × w" },
    { left: "Area of triangle",  right: "½ × b × h" },
  ],
}
// SPOT_ERROR / STEP — additional fields
{
  subtype: "STEP",
  steps: [
    { id: 1, text: "a² + b² = c²",          isError: false },
    { id: 4, text: "c² = 24",               isError: true  },  // ← exactly one error
  ],
}
// SPOT_ERROR / VALUE — additional fields
{
  subtype:          "VALUE",
  expression:       "A = [π|1] × [r²|2] = [50.24|3] cm²",
  correctErrorId:   3,
  errorExplanation: "The radius is 4 cm, so r² = 16. The correct area is π × 16 ≈ 50.27 cm².",
}
// EXPLANATION — additional fields
{
  modelAnswer:      "Because the relationship is linear…",
  markingChecklist: [
    "Identifies the relationship as linear",
    "States that the scale factor is 2",
    "Gives a correct numerical example",
  ],
}
```

## QUESTION PLANNING RULES

**Type Mix:**
Apply `question_type_mix` proportionally within each level. For a level with 10 questions
and a mix of 30% NUMERIC / 20% MCQ / 15% MATCH / 20% SPOT_ERROR / 15% EXPLANATION,
plan approximately 3 NUMERIC, 2 MCQ, 2 MATCH, 2 SPOT_ERROR, 1 EXPLANATION. Round
sensibly — the mix is a guideline, not a strict quota.

**Difficulty Distribution:**
Spread difficulties 1–4 across the `questions_per_level` questions within each level.
Do not plan all questions at the same difficulty.

**Skill Fidelity:**
Each level's questions must test the single skill defined for that level in the Topic
Analysis. Do not let questions drift into testing skills from other levels.

**Context Problems:**
At higher skill levels (Context Problem tier), wrap the question in a real-world NZ
narrative. Use the context themes from Section 4 of the Topic Analysis. The mathematical
structure remains NUMERIC or EXPLANATION.

**Terminology / Recall:**
At Levels 1–2 (Pre-req Check, Facts), MCQ and MATCH questions may test mathematical
vocabulary: correct spelling of terms, matching terms to definitions, identifying symbols.
These are valid and pedagogically important (Rosenshine: retrieval practice).

**Example Problems:**
Use the example problems in Section 2 for style calibration — register, sentence
structure, number ranges, diagram style. Do NOT copy any question verbatim.

**MCQ Distractors:**
All four MCQ options must be computed values derived from the question parameters.
Placeholder strings such as "incorrect option A" are never acceptable. For fractions,
distractors must represent operations on the numerator or denominator separately. For
surds, distractors must represent halving, doubling, or squaring the radicand. Distractors
must represent genuine student misconceptions from the Topic Analysis.

**MATCH Pairs:**
Pairs must be noted as "will be shuffled" in the Dev note. The left and right columns in
the Review Document may be shown aligned for readability, but the JS module must shuffle
them before output.

**SPOT_ERROR:**
Exactly ONE step or token must contain the error. Write the error explanation in plain
English so the teacher can verify it without running code.

**EXPLANATION:**
EXPLANATION questions must ask students to EXPLAIN, JUSTIFY, or DESCRIBE — not merely
calculate. The model answer must be in clear, NCEA-appropriate mathematical language.
The marking checklist must contain 2–4 items, each independently verifiable with no
overlap between items.

**Number Ranges:**
Choose number ranges that produce clean answers appropriate for the year group. Avoid
ugly recurring decimals at low difficulty levels. State the ranges explicitly in the
Dev note for every NUMERIC question.

**LaTeX:**
All mathematical expressions in question text and working must use LaTeX notation.
In this document, render LaTeX with single backslashes for readability: \frac{1}{2}.
In the Dev note, explicitly flag that the JS string will require double backslashes:
\\frac{1}{2}.
